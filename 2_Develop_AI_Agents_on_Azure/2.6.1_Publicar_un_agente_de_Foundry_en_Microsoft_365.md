# Publicar un agente de Foundry en Microsoft Teams

## Requisitos previos

Antes de publicar el agente, Microsoft recomienda:

### 1. Probar el agente

Validar en el **Foundry Playground** que:

- Responde correctamente.
- Las herramientas (Tools) funcionan.
- Las respuestas son apropiadas para el caso de uso.

---

### 2. Verificar permisos

Se requieren los siguientes roles:

| Rol | Propósito |
|------|-----------|
| **Azure AI Project Manager** | Publicar agentes |
| **Azure AI User** | Invocar o conversar con el agente publicado |
| Permisos sobre la suscripción Azure | Crear recursos |
| Permisos en Microsoft Entra ID | Registrar aplicaciones |

---

### 3. Registrar Azure Bot Service

El proceso crea automáticamente un **Azure Bot Service**.

Antes de publicar, verifica que el proveedor:

```text
Microsoft.BotService
```

esté registrado en la suscripción de Azure.

---

### 4. Preparar la metadata

Debes proporcionar:

- Nombre del agente.
- Descripción.
- Icono 32×32 px.
- Icono 192×192 px.
- Información del publicador.
- URL de Política de privacidad.
- URL de Términos de uso.

> **Importante:** Nunca incluyas secretos, API Keys o información sensible en estos campos.

---

# Proceso de publicación

## Paso 1. Seleccionar el agente

En **Microsoft AI Foundry**:

- Abrir el proyecto.
- Seleccionar el agente.
- Verificar que la configuración sea correcta.

---

## Paso 2. Iniciar la publicación

Seleccionar:

```text
Publish
    ↓
Publish
    ↓
Publish to Teams and Microsoft 365 Copilot
```

Se abrirá el asistente de publicación.

---

## Paso 3. Configurar Azure Bot Service

Durante el proceso:

- Se genera automáticamente un **Application ID**.
- Se genera un **Tenant ID**.
- Se crea un nuevo **Azure Bot Service**.

Estos identificadores son útiles para diagnóstico y solución de problemas.

---

## Paso 4. Completar la metadata

Completar:

| Campo | Descripción |
|--------|-------------|
| Name | Nombre visible del agente |
| Description | Descripción corta |
| Icons | Iconos PNG |
| Publisher | Organización y contacto |
| Privacy Policy | URL |
| Terms of Use | URL |

---

## Paso 5. Elegir el alcance (Publish Scope)

### Shared Scope

- Disponible inmediatamente.
- Aparece en **Your agents**.
- No requiere aprobación del administrador.

Ideal para:

- Desarrollo.
- Pruebas.
- Equipos pequeños.

---

### Organization Scope

- Aparece en **Built by your org**.
- Requiere aprobación del administrador de Microsoft 365.

Ideal para:

- Producción.
- Toda la organización.

---

## Paso 6. Preparar el paquete

Seleccionar:

```text
Prepare Agent
```

El portal genera un paquete de publicación (.zip).

Luego puedes:

- Descargar el paquete para pruebas.
- Continuar con la publicación directa.

---

# Probar el paquete en Teams

Si descargaste el paquete:

```text
Teams
    ↓
Apps
    ↓
Manage your apps
    ↓
Upload a custom app
```

Selecciona el archivo **.zip** generado.

Verifica que:

- ✅ El agente responde.
- ✅ Las respuestas son correctas.
- ✅ El tiempo de respuesta es adecuado.
- ✅ Las Tools funcionan correctamente.

---

# Aprobación para Organization Scope

Para publicar en toda la organización, un administrador debe aprobar el agente desde:

```text
Microsoft 365 Admin Center
    ↓
Agents
    ↓
All
    ↓
Requested
```

Después de aprobarlo:

```text
Approve request and activate
```

El agente aparecerá en:

```text
Built by your org
```

para todos los usuarios autorizados.

---

# Reasignar permisos

Al publicar, **Foundry crea una nueva identidad para el agente**.

Si el agente utiliza recursos de Azure (por ejemplo, Azure AI Search o Storage), es necesario asignar nuevamente los permisos RBAC a esta nueva identidad.

De lo contrario:

- Las Tools que funcionaban durante el desarrollo pueden dejar de funcionar en producción.

---

# Actualizar un agente publicado

Cuando se realizan cambios:

1. Modificar el agente en Foundry.
2. Probar los cambios.
3. Volver a publicar.
4. Generar un nuevo paquete.
5. Desplegar la nueva versión.

Dependiendo del **Publish Scope**, la actualización puede requerir nuevamente la aprobación del administrador.

---

# Flujo completo

```text
Desarrollar agente
        │
        ▼
Probar en Playground
        │
        ▼
Verificar permisos
        │
        ▼
Configurar metadata
        │
        ▼
Crear Azure Bot Service
        │
        ▼
Elegir Publish Scope
        │
        ▼
Generar paquete (.zip)
        │
        ▼
Probar en Teams
        │
        ▼
(Aprobación del administrador si aplica)
        │
        ▼
Agente disponible para los usuarios
```

---

# Resumen

- Antes de publicar, prueba el agente y verifica los permisos necesarios.
- El proceso de publicación crea automáticamente un **Azure Bot Service** y una **nueva identidad** para el agente.
- Existen dos alcances:
  - **Shared Scope:** pruebas y equipos pequeños.
  - **Organization Scope:** producción con aprobación del administrador.
- Tras publicar, es importante **reasignar los permisos RBAC** a la nueva identidad para que las herramientas sigan funcionando.
- Cada modificación del agente requiere **volver a publicar** una nueva versión.
