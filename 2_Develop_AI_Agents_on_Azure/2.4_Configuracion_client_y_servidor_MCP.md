# Configuración de un cliente y servidor MCP

## Arquitectura MCP

El **Model Context Protocol (MCP)** sigue una arquitectura cliente-servidor para permitir que los agentes de IA utilicen herramientas externas de forma estandarizada.

Cada componente tiene una responsabilidad específica:

```text
Usuario
    │
    ▼
Aplicación (Host)
    │
    ▼
MCP Client
    │
    ▼
MCP Server
    │
    ▼
Herramientas / APIs / Bases de datos
```

- **Host:** aplicación que contiene al agente (por ejemplo, AI Foundry o VS Code).
- **MCP Client:** componente que implementa el protocolo MCP y se comunica con los servidores.
- **MCP Server:** expone herramientas, recursos y prompts que el agente puede utilizar.

---

# MCP Client

El **MCP Client** es el intermediario entre el agente y los servidores MCP.

Sus responsabilidades son:

- Descubrir las herramientas disponibles.
- Enviar solicitudes al servidor.
- Recibir los resultados.
- Entregar la respuesta al agente.

Un cliente puede conectarse a **uno o varios servidores MCP** dependiendo de la aplicación.

---

# MCP Server

El **MCP Server** publica capacidades que otros clientes pueden consumir.

Puede exponer:

- **Tools**: funciones que ejecutan acciones.
- **Resources**: información o documentos.
- **Prompts**: plantillas reutilizables para el modelo.

Cada servidor describe estas capacidades mediante metadatos para que el cliente pueda descubrirlas automáticamente. :contentReference[oaicite:0]{index=0}

---

# Configuración del servidor

Para que un cliente pueda utilizar un servidor MCP, debe configurarse la conexión.

La configuración normalmente incluye:

- Nombre del servidor.
- Tipo de transporte.
- Dirección (URL o comando local).
- Credenciales (si son necesarias).

Ejemplo conceptual:

```text
Servidor:
- Nombre: Inventory Server
- Transporte: HTTP
- URL: https://...
```

---

# Tipos de transporte

MCP admite distintos mecanismos de comunicación entre cliente y servidor.

## STDIO

El servidor se ejecuta como un proceso local y se comunica mediante la entrada y salida estándar.

```text
Host
   │
   ▼
MCP Client
   │
 STDIO
   │
   ▼
Servidor local
```

**Ideal para:**

- Desarrollo local.
- Herramientas instaladas en el equipo.
- Baja latencia.

---

## HTTP (Remote MCP)

El servidor se ejecuta de forma remota y el cliente se comunica mediante HTTP.

```text
Host
   │
   ▼
MCP Client
   │
 HTTP
   │
   ▼
Servidor remoto
```

**Ideal para:**

- Servicios compartidos.
- Aplicaciones empresariales.
- Infraestructura en la nube.

---

# Flujo de conexión

1. El Host inicia el MCP Client.
2. El cliente establece conexión con el servidor.
3. Solicita las herramientas disponibles (**Tool Discovery**).
4. El servidor devuelve sus capacidades.
5. El agente utiliza la herramienta cuando la necesita.
6. El servidor ejecuta la acción y devuelve el resultado.

---

# Beneficios de esta arquitectura

- Separación clara entre cliente y servidor.
- Herramientas reutilizables por distintos agentes.
- Integración sencilla con nuevos servicios.
- Comunicación estandarizada mediante MCP.
- Posibilidad de combinar múltiples servidores en una misma aplicación.

---

# Resumen

- Un sistema MCP está compuesto por **Host**, **MCP Client** y **MCP Server**.
- El **Host** ejecuta el agente y utiliza un **MCP Client** para comunicarse con servidores MCP.
- El **MCP Server** publica herramientas, recursos y prompts que pueden ser descubiertos automáticamente.
- Los servidores pueden conectarse mediante **STDIO** (local) o **HTTP** (remoto).
- Esta arquitectura facilita integrar agentes con herramientas externas de forma estandarizada y reutilizable.
