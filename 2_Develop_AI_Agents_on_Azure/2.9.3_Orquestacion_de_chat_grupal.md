# Group Chat Orchestration

## ¿Qué es?

La **Group Chat Orchestration** permite que **múltiples agentes (y opcionalmente un humano)** colaboren mediante una conversación compartida.

Un **Chat Manager** controla el flujo de la conversación, decidiendo:

- Qué agente habla a continuación.
- Cuándo solicitar intervención humana.
- Cuándo finalizar la conversación.

```text
                 Usuario
                    │
                    ▼
             Chat Manager
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
 Agente A      Agente B      Agente C
      ▲             ▲             ▲
      └─────────────┴─────────────┘
             Conversación compartida
```

---

# Características

- Conversación colaborativa entre múltiples agentes.
- Todos los participantes comparten el mismo contexto.
- Un **Chat Manager** coordina los turnos.
- Puede incluir intervención humana (*Human-in-the-loop*).
- Todas las respuestas quedan registradas en un único hilo.

---

# ¿Cuándo utilizar Group Chat?

Es recomendable cuando se necesita:

- Colaboración entre varios agentes.
- Discusión iterativa.
- Supervisión humana.
- Transparencia sobre cómo se llegó a una decisión.

### Casos de uso

- Brainstorming.
- Debates.
- Resolución colaborativa de problemas.
- Toma de decisiones por consenso.
- Validación de respuestas.
- Procesos de creación y revisión de contenido.

---

# ¿Cuándo evitarlo?

No es recomendable cuando:

- Un pipeline secuencial es suficiente.
- Se requiere la máxima velocidad.
- El flujo debe ser totalmente determinista.
- El Chat Manager no puede determinar cuándo finalizar.
- Hay demasiados agentes (se recomienda **3 o menos** para facilitar la coordinación).

---

# Maker-Checker Loop

Uno de los patrones más comunes dentro de Group Chat.

Un agente crea contenido (**Maker**) y otro lo revisa (**Checker**).

```text
Maker
   │
   ▼
Contenido
   │
   ▼
Checker
   │
¿Correcto?
 ├── Sí → Finalizar
 └── No → Retroalimentación
             │
             ▼
          Maker
```

Este ciclo continúa hasta obtener un resultado satisfactorio.

Ideal para:

- Revisión de documentos.
- Generación de código.
- Validación de contenido.
- Control de calidad.

# Implementación

## 1. Crear el Chat Client

Conectar la aplicación al proveedor de IA.

Ejemplo:

```text
AzureOpenAIChatClient
```

---

## 2. Definir los agentes

Crear los agentes participantes.

Cada agente debe tener:

- Nombre.
- Instrucciones.
- Especialización.

---

## 3. Construir el Workflow

Utilizar:

```text
GroupChatBuilder
```

Agregar los participantes:

```text
participants(...)
```

Construir el workflow:

```text
build()
```

---

## 4. Ejecutar el Workflow

Ejecutar la conversación mediante:

```text
workflow.run(...)
```

---

## 5. Obtener resultados

Recuperar las respuestas con:

```text
get_outputs()
```

Cada mensaje incluye:

- Autor.
- Contenido.

---

## 6. Procesar la conversación

La aplicación puede:

- Mostrar toda la conversación.
- Generar un resumen.
- Extraer la respuesta final.
- Analizar quién aportó cada idea.

---

# Personalizar el Chat Manager

Es posible crear un administrador personalizado heredando de:

```text
GroupChatManager
```

Permite controlar:

- Cómo resumir los resultados.
- Qué agente habla después.
- Cuándo solicitar intervención humana.
- Cuándo finalizar la conversación.

Esto permite adaptar la dinámica del grupo al caso de uso.

---

# Orden de ejecución del Chat Manager

En cada ronda, el Chat Manager ejecuta los siguientes métodos:

## 1. should_request_user_input

Determina si es necesario pedir información al usuario antes de continuar.

---

## 2. should_terminate

Decide si la conversación debe finalizar.

Ejemplos:

- Se alcanzó el número máximo de rondas.
- El objetivo ya fue cumplido.

---

## 3. filter_results

Si la conversación termina:

- Resume.
- Filtra.
- Procesa el resultado final.

---

## 4. select_next_agent

Si continúa la conversación:

Selecciona el siguiente agente que participará.

# Flujo completo

```text
Crear Chat Client
        │
        ▼
Crear Agentes
        │
        ▼
GroupChatBuilder
        │
        ▼
Agregar participantes
        │
        ▼
build()
        │
        ▼
workflow.run()
        │
        ▼
Chat Manager
        │
        ▼
Selecciona siguiente agente
        │
        ▼
Conversación colaborativa
        │
        ▼
get_outputs()
        │
        ▼
Respuesta final
```

---

# Ventajas

- Favorece la colaboración.
- Permite combinar distintas especialidades.
- Transparencia gracias al historial completo.
- Compatible con Human-in-the-loop.
- Muy flexible para problemas abiertos.

---

# Limitaciones

- Mayor tiempo de ejecución.
- Más consumo de tokens.
- Requiere un Chat Manager bien diseñado.
- Puede volverse difícil de controlar con muchos agentes.

---

# Comparación

| Sequential | Group Chat |
|------------|------------|
| Pipeline fijo | Conversación colaborativa |
| Orden predefinido | Turnos dinámicos |
| Sin debate | Debate e intercambio de ideas |
| Ideal para procesos | Ideal para colaboración |

---

# Resumen

- **Group Chat Orchestration** coordina una conversación entre múltiples agentes (y opcionalmente humanos) mediante un **Chat Manager**.
- El Chat Manager controla quién participa, cuándo solicitar intervención humana y cuándo finalizar la conversación.
- Es ideal para **brainstorming, debates, validación, consenso y resolución colaborativa de problemas**.
- Se implementa con **GroupChatBuilder**, agregando los agentes con `participants()` y ejecutando el workflow con `run()`.
- Puede personalizarse extendiendo la clase **GroupChatManager**, modificando la selección de agentes, el manejo de resultados y las condiciones de terminación.
