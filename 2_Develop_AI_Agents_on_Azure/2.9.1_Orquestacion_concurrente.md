# Concurrent Orchestration

## ¿Qué es?

La **Concurrent Orchestration** ejecuta **múltiples agentes en paralelo** sobre una misma tarea.

Cada agente trabaja de forma **independiente**, sin compartir información con los demás, y posteriormente sus resultados pueden combinarse para producir una respuesta final.

```text
                Solicitud
                    │
                    ▼
        ┌───────────┼───────────┐
        ▼           ▼           ▼
    Agente A    Agente B    Agente C
        │           │           │
        └───────────┼───────────┘
                    ▼
        Agregación de resultados
                    │
                    ▼
            Respuesta final
```

---

# Características

- Todos los agentes trabajan **simultáneamente**.
- Cada agente utiliza sus propias instrucciones y especialización.
- Los agentes **no comparten contexto** entre sí.
- La ejecución paralela reduce el tiempo de procesamiento.
- Los resultados pueden combinarse o mantenerse independientes.

> Un agente puede ejecutar su propia orquestación interna si necesita colaborar con otros agentes.

---

# ¿Cuándo utilizar Concurrent Orchestration?

Es recomendable cuando:

- Las tareas pueden ejecutarse en paralelo.
- Se desea obtener diferentes perspectivas sobre un mismo problema.
- Se necesitan especialistas trabajando de forma independiente.
- El tiempo de respuesta es importante.

### Casos de uso

- Brainstorming.
- Ensemble Reasoning.
- Votación o consenso (*Quorum*).
- Análisis paralelos.
- Comparación de soluciones.
- Ejecución simultánea de herramientas.

---

# ¿Cuándo evitarla?

No es recomendable cuando:

- Los agentes dependen del trabajo de otros.
- El proceso requiere un orden específico.
- Existen restricciones de recursos o cuotas del modelo.
- Varios agentes modifican el mismo recurso simultáneamente.
- Es difícil resolver conflictos entre respuestas.
- Combinar resultados disminuye la calidad de la solución.

En estos casos suele ser más apropiado utilizar **Sequential** o **Handoff Orchestration**.

# Implementación

## 1. Crear el Chat Client

Conectar la aplicación al proveedor de IA (por ejemplo, Azure OpenAI).

```text
AzureOpenAIChatClient
```

---

## 2. Definir los agentes

Crear varios agentes especializados.

Ejemplo:

- Agente Técnico.
- Agente de Negocio.
- Agente Creativo.

Cada uno debe tener:

- Nombre.
- Instrucciones.
- Especialización.

---

## 3. Construir el Workflow

Utilizar **ConcurrentBuilder**.

Agregar todos los agentes mediante:

```text
participants(...)
```

Finalmente construir el workflow con:

```text
build()
```

---

## 4. Ejecutar el Workflow

Ejecutar el workflow enviando la tarea.

Todos los agentes comienzan a trabajar al mismo tiempo.

```text
workflow.run(...)
```

---

## 5. Obtener resultados

Recuperar las respuestas mediante:

```text
get_outputs()
```

Cada salida incluye:

- Autor (agente).
- Contenido.

---

## 6. Procesar las respuestas

La aplicación puede:

- Combinar las respuestas.
- Compararlas.
- Elegir la mejor.
- Presentarlas individualmente.

---

# Flujo completo

```text
Crear Chat Client
        │
        ▼
Crear Agentes
        │
        ▼
ConcurrentBuilder
        │
        ▼
Agregar participantes
        │
        ▼
build()
        │
        ▼
workflow.run()
        │
        ▼
Todos los agentes trabajan en paralelo
        │
        ▼
get_outputs()
        │
        ▼
Respuesta agregada
```

---

# Ventajas

- Alta velocidad gracias a la ejecución paralela.
- Aprovecha distintas especializaciones.
- Produce respuestas más diversas.
- Mejora la toma de decisiones mediante múltiples perspectivas.
- Escalable para problemas complejos.

---

# Limitaciones

- Los agentes no comparten contexto.
- Puede consumir más recursos.
- Requiere un mecanismo para combinar respuestas.
- No es adecuado para tareas con dependencias entre pasos.

---

# Resumen

- **Concurrent Orchestration** ejecuta múltiples agentes **en paralelo** sobre una misma tarea.
- Cada agente trabaja de forma independiente y sus resultados pueden combinarse o utilizarse por separado.
- Es ideal para **brainstorming, análisis paralelos, ensemble reasoning y decisiones por consenso**.
- Se implementa mediante **ConcurrentBuilder**, agregando los agentes participantes y ejecutando el workflow con `run()`.
- Debe utilizarse únicamente cuando las tareas puedan ejecutarse de forma independiente y la coordinación entre agentes no sea necesaria.
