# Prompt Engineering

## ¿Qué es?

**Prompt Engineering** es el proceso de diseñar y optimizar instrucciones (*prompts*) para obtener mejores resultados de un modelo generativo.

El objetivo es guiar al modelo para que genere respuestas más precisas, útiles y consistentes. :contentReference[oaicite:0]{index=0}

---

# ¿Por qué es importante?

Los modelos generativos pueden producir respuestas muy distintas dependiendo de cómo se formule la solicitud.

Un buen prompt ayuda a:

- Mejorar la calidad de las respuestas.
- Reducir ambigüedades.
- Obtener formatos específicos.
- Aumentar la consistencia.
- Adaptar el comportamiento del modelo al caso de uso. :contentReference[oaicite:1]{index=1}

---

# Componentes de un prompt efectivo

## 1. Instrucciones

Indican claramente qué debe hacer el modelo.

**Ejemplo**

```text
Resume el siguiente artículo en 5 viñetas.
```

---

## 2. Contexto

Información adicional que ayuda al modelo a comprender la situación.

**Ejemplo**

```text
Eres un analista financiero.
Explica este reporte para ejecutivos sin conocimientos técnicos.
```

---

## 3. Datos de entrada

Información sobre la que debe trabajar el modelo.

**Ejemplo**

```text
Analiza las siguientes ventas:
...
```

---

## 4. Formato de salida

Define cómo debe presentarse la respuesta.

**Ejemplo**

```text
Devuelve la respuesta en formato JSON.
```

# Patrones de Prompt Engineering

## 1. Persona Pattern

Consiste en indicar al modelo el **rol o perspectiva** que debe adoptar antes de responder.

Esto influye en el tono, vocabulario y nivel de detalle de la respuesta.

**Ejemplo**

```text
System:
Eres un profesional de marketing con experiencia en productos tecnológicos.

User:
Escribe una descripción de una línea para un CRM.
```

### Casos de uso

- Profesor.
- Médico.
- Desarrollador senior.
- Analista financiero.
- Traductor.
- Experto en ciberseguridad.

> Definir una persona suele producir respuestas más especializadas y consistentes.

---

## 2. Format Template Pattern

Consiste en indicar explícitamente el **formato** que debe tener la respuesta.

Esto facilita procesar la salida automáticamente.

**Ejemplo**

```text
Devuelve la información con el siguiente formato:

- Nombre del hotel
- Ubicación
- Número de estrellas
- Precio por noche
```

### Casos de uso

- JSON.
- XML.
- Markdown.
- Tablas.
- Listas.
- Informes estructurados.

---

## 3. Chain-of-Thought Pattern

Se pide al modelo resolver un problema **paso a paso** antes de responder.

**Ejemplo**

```text
¿Cuál es el mejor hotel para una familia de cuatro personas?

Analiza paso a paso:

1. Tamaño de la habitación.
2. Servicios para niños.
3. Ubicación.
4. Precio.
```

### Variante: Descomposición de tareas

En lugar de pedir todo en un solo prompt, se divide el problema en varias etapas.

Ejemplo:

```text
Prompt 1:
Extrae los hechos importantes del documento.

↓

Prompt 2:
Responde la pregunta usando únicamente esos hechos.
```

Esta estrategia reduce errores en tareas complejas.

> **Nota:** Este patrón está pensado para modelos **no razonadores**. Modelos de razonamiento (como la serie **o**) realizan este proceso internamente.

---

## 4. Few-shot Learning Pattern

Consiste en proporcionar uno o varios ejemplos para que el modelo aprenda el patrón esperado.

### One-shot

Se proporciona un único ejemplo.

```text
Mensaje:
Quiero cambiar mi vuelo.

Categoría:
Cambio de reserva

Mensaje:
Necesito cancelar mi hotel.

Categoría:
```

### Few-shot

Se proporcionan varios ejemplos.

```text
Mensaje:
Quiero cambiar mi vuelo.

Categoría:
Cambio de reserva

Mensaje:
¿Cómo estará el clima en París?

Categoría:
Información turística

Mensaje:
Quiero solicitar un reembolso.

Categoría:
```

### Zero-shot

No se proporciona ningún ejemplo; únicamente se describe la tarea.

```text
Clasifica el siguiente mensaje del cliente.
```

---

## 5. Clear Syntax and Delimiters Pattern

Cuando el prompt contiene varias secciones (instrucciones, contexto, ejemplos, documentos, etc.), es recomendable separarlas mediante delimitadores.

Esto ayuda al modelo a distinguir claramente cada parte del prompt y reduce interpretaciones incorrectas.

### Delimitadores comunes

- `---`
- Encabezados Markdown (`#`, `##`)
- Triple comilla (```)
- Etiquetas XML

**Ejemplo**

````text
# Instrucciones

Resume el documento en cinco puntos.

---

# Documento

...

---
# Formato esperado

- Punto 1
- Punto 2
- Punto 3
````

# Parámetros del modelo

Además del prompt, el comportamiento puede ajustarse mediante parámetros.

## Temperature

Controla la aleatoriedad.

### Baja temperatura (0.0 - 0.3)

- Más determinista.
- Respuestas consistentes.
- Menor creatividad.

Ideal para:

- Clasificación.
- Extracción de datos.
- Código.

---

### Alta temperatura (0.7 - 1.0)

- Más creatividad.
- Más variedad.
- Mayor riesgo de errores.

Ideal para:

- Escritura creativa.
- Brainstorming.
- Generación de ideas.

---

## Max Tokens

Define la longitud máxima de la respuesta.

- Mayor valor → respuestas más largas.
- Menor valor → respuestas más breves.

# Buenas prácticas

## Sé específico

❌ Malo

```text
Háblame de bases de datos.
```

✅ Mejor

```text
Explica las diferencias entre PostgreSQL y MongoDB para una aplicación SaaS.
```

---

## Define el formato esperado

❌

```text
Analiza estos datos.
```

✅

```text
Analiza estos datos y devuelve:
1. Hallazgos principales.
2. Riesgos.
3. Recomendaciones.
```

---

## Proporciona contexto

Mientras más relevante sea el contexto, mejores serán los resultados.

```text
Eres un arquitecto de software.
```

---

## Usa ejemplos cuando sea necesario

Los ejemplos ayudan al modelo a entender exactamente qué se espera.

---

# Limitaciones del Prompt Engineering

No todos los problemas se resuelven mejorando prompts.

Cuando se requiere:

- Información privada.
- Conocimiento empresarial.
- Datos actualizados.
- Comportamientos altamente especializados.

Puede ser mejor utilizar:

- **RAG (Retrieval-Augmented Generation)**.
- **Fine-Tuning**.
- **Tools** (Web Search, File Search, Function Calling). :contentReference[oaicite:5]{index=5}

---

# Resumen

- **Prompt Engineering** consiste en diseñar instrucciones para optimizar la salida de un modelo.
- Los componentes clave son:
  - Instrucciones.
  - Contexto.
  - Datos de entrada.
  - Formato de salida.
- Las técnicas principales son:
  - **Zero-shot prompting**.
  - **Few-shot prompting**.
  - **System messages**.
- Parámetros como **Temperature** y **Max Tokens** también influyen en el resultado.
- Es la primera estrategia para optimizar un modelo antes de recurrir a RAG o Fine-Tuning. :contentReference[oaicite:6]{index=6}
