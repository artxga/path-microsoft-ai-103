# Sequential Orchestration

## ¿Qué es?

La **Sequential Orchestration** organiza los agentes en un **pipeline**, donde cada agente procesa la salida del anterior.

El resultado de un agente se convierte en la entrada del siguiente, permitiendo refinar progresivamente la información.

```text
Solicitud
    │
    ▼
Agente 1
    │
    ▼
Agente 2
    │
    ▼
Agente 3
    │
    ▼
Respuesta final
```

---

# Características

- Los agentes se ejecutan **uno tras otro**.
- Cada agente depende del resultado del anterior.
- El orden está definido previamente.
- Los agentes **no deciden** cuál será el siguiente paso.
- Produce resultados consistentes y predecibles.

---

# ¿Cuándo utilizar Sequential Orchestration?

Es recomendable cuando:

- El proceso tiene varias etapas dependientes.
- Cada paso agrega información necesaria para el siguiente.
- Las tareas deben ejecutarse en un orden específico.
- Se busca mejorar progresivamente un resultado.

### Casos de uso

- Revisión de documentos.
- Transformación de datos.
- Creación de contenido.
- Pipelines de procesamiento.
- Razonamiento en múltiples etapas.
- Draft → Review → Improve.

---

# ¿Cuándo evitarla?

No es recomendable cuando:

- Las tareas pueden ejecutarse en paralelo.
- Un solo agente puede resolver todo el problema.
- Se requiere colaboración dinámica entre agentes.
- El flujo necesita retroceder o cambiar de dirección.
- El procesamiento depende de decisiones tomadas durante la ejecución.

En estos casos suelen ser mejores patrones como **Concurrent**, **Handoff** o **Group Chat**.

---

# Implementación

## 1. Crear el Chat Client

Configurar la conexión con el proveedor de IA.

Ejemplo:

```text
AzureOpenAIChatClient
```

---

## 2. Definir los agentes

Crear los agentes que participarán en el pipeline.

Cada agente debe tener:

- Nombre.
- Instrucciones.
- Rol específico.

Ejemplo:

- Agente Redactor.
- Agente Revisor.
- Agente Corrector.

---

## 3. Construir el Workflow

Utilizar **SequentialBuilder**.

Agregar los agentes mediante:

```text
participants(...)
```

Construir el workflow:

```text
build()
```

---

## 4. Ejecutar el Workflow

Ejecutar el pipeline utilizando:

```text
run_stream(...)
```

Cada agente recibe automáticamente la salida del anterior.

---

## 5. Procesar eventos

Durante la ejecución pueden recibirse eventos del workflow.

El evento más importante es:

```text
WorkflowOutputEvent
```

que contiene el resultado generado en cada etapa.

---

## 6. Obtener la conversación final

Al finalizar, se obtiene la conversación completa, mostrando cómo cada agente contribuyó al resultado.

---

---

# Ventajas

- Flujo simple y fácil de entender.
- Resultados predecibles.
- Refinamiento progresivo de la información.
- Adecuado para procesos estructurados.
- Fácil de depurar.

---

# Limitaciones

- No aprovecha el paralelismo.
- La velocidad depende del agente más lento.
- Un error temprano puede afectar todas las etapas posteriores.
- No permite colaboración dinámica entre agentes.

---

# Comparación rápida

| Concurrent | Sequential |
|------------|------------|
| Agentes trabajan en paralelo | Agentes trabajan en orden |
| Máxima velocidad | Máximo control |
| Resultados independientes | Resultados dependientes |
| Ideal para análisis paralelos | Ideal para pipelines |

---

# Resumen

- **Sequential Orchestration** ejecuta los agentes **de forma secuencial**, donde la salida de un agente es la entrada del siguiente.
- Es el patrón ideal para **pipelines**, **procesamiento por etapas**, **transformación de datos** y **refinamiento progresivo**.
- Se implementa mediante **SequentialBuilder**, agregando los agentes con `participants()` y ejecutando el workflow con `run_stream()`.
- Debe utilizarse cuando el orden de ejecución es importante y cada etapa depende de la anterior.
