# Magentic Orchestration

## ¿Qué es?

La **Magentic Orchestration** es un patrón de múltiples agentes diseñado para resolver **problemas complejos y abiertos**, donde no existe un flujo de trabajo definido desde el inicio.

Un **Magentic Manager** coordina dinámicamente un conjunto de agentes especializados, decidiendo:

- Qué agente debe intervenir.
- Qué subtarea debe ejecutarse.
- Cómo evoluciona el plan de trabajo.
- Cuándo finalizar.

A diferencia de otros patrones, el flujo **no está predefinido**, sino que se construye conforme avanza la ejecución.

```text
                  Solicitud
                      │
                      ▼
             Magentic Manager
                      │
      ┌───────────────┼───────────────┐
      ▼               ▼               ▼
 Agente A        Agente B        Agente C
      ▲               ▲               ▲
      └───────────────┼───────────────┘
                      │
          Ledger (plan dinámico)
                      │
                      ▼
              Respuesta final
```

---

# Características

- El flujo se adapta durante la ejecución.
- Existe un **Manager** que coordina todo el proceso.
- Los agentes colaboran según el contexto.
- Se genera un **Task Ledger**, que documenta objetivos, subtareas y progreso.
- Está pensado para problemas donde no se conoce la solución desde el inicio.

---

# ¿Qué es el Task Ledger?

El **Task Ledger** es un registro dinámico que mantiene el estado del problema durante la ejecución.

Incluye:

- Objetivos.
- Subobjetivos.
- Plan de ejecución.
- Progreso.
- Decisiones tomadas.

Este registro se actualiza continuamente mientras los agentes colaboran.

```text
Objetivo
    │
    ├── Subtarea A ✔
    ├── Subtarea B ✔
    ├── Subtarea C...
    └── Próximo paso
```

El Ledger facilita:

- Seguimiento del progreso.
- Planificación.
- Auditoría.
- Revisión humana.

---

# ¿Cuándo utilizar Magentic Orchestration?

Es recomendable cuando:

- El problema es complejo.
- No existe un flujo fijo.
- Se requiere planificación dinámica.
- Se necesita colaboración entre especialistas.
- Es importante documentar el proceso de resolución.
- Los agentes utilizan herramientas que interactúan con sistemas externos.

### Casos de uso

- Investigación compleja.
- Planeación estratégica.
- Automatización empresarial.
- Diagnóstico de problemas.
- Resolución de incidentes.
- Sistemas de soporte inteligentes.

---

# ¿Cuándo evitarla?

No es recomendable cuando:

- El flujo ya está definido.
- El problema es sencillo.
- No se necesita un plan de ejecución.
- La prioridad es la velocidad.
- Existe riesgo de ciclos o bloqueos prolongados.

En estos casos suelen ser mejores **Sequential** o **Concurrent**.

# Implementación

## 1. Definir agentes especializados

Crear varios agentes con diferentes responsabilidades.

Ejemplo:

- Investigador.
- Analista.
- Programador.
- Revisor.

Cada uno debe tener:

- Nombre.
- Instrucciones.
- Herramientas.

---

## 2. Configurar el Event Callback

Crear una función asíncrona para recibir los eventos del workflow.

Permite manejar:

- Mensajes del Manager.
- Mensajes de los agentes.
- Actualizaciones en streaming.
- Resultado final.

---

## 3. Construir el Workflow

Utilizar:

```text
MagenticBuilder
```

Agregar los agentes participantes y configurar:

- Callback.
- Streaming.
- Manager.

---

## 4. Configurar el Standard Manager

El Manager controla la planificación y coordinación.

Puede configurarse con parámetros como:

- **max_round_count** → número máximo de rondas.
- **stall_count** → límite de bloqueos antes de detener el proceso.
- **reset_count** → reinicios permitidos.

---

## 5. Ejecutar el Workflow

Ejecutar la tarea mediante:

```text
run_stream(...)
```

Durante la ejecución el Manager:

- Planifica.
- Divide tareas.
- Selecciona agentes.
- Actualiza el Ledger.
- Replanifica cuando sea necesario.

---

## 6. Procesar eventos

Durante la ejecución se reciben eventos del workflow.

El evento más importante es:

```text
WorkflowOutputEvent
```

que contiene el resultado final.

---

## 7. Obtener la respuesta

Al finalizar se obtiene:

- Solución final.
- Conversación.
- Plan ejecutado.
- Historial de colaboración.

---

# Ventajas

- Muy flexible.
- Excelente para problemas abiertos.
- Planificación dinámica.
- Documenta todo el proceso mediante el **Task Ledger**.
- Aprovecha múltiples especialistas.
- Puede adaptarse durante la ejecución.

---

# Limitaciones

- Más complejo que otros patrones.
- Mayor consumo de tokens.
- Más lento debido a la planificación.
- Requiere una buena configuración del Manager.
- Puede entrar en ciclos si no se establecen límites.

---

# Comparación

| Sequential | Group Chat | Magentic |
|------------|------------|-----------|
| Flujo fijo | Conversación | Planificación dinámica |
| Pipeline | Debate | Delegación inteligente |
| Sin planificación | Chat Manager | Magentic Manager |
| Procesos definidos | Colaboración | Problemas abiertos |

---

# Resumen

- **Magentic Orchestration** es un patrón de múltiples agentes para resolver **problemas complejos y abiertos**, donde el plan de ejecución se construye dinámicamente.
- Un **Magentic Manager** coordina la colaboración, decide qué agente actúa en cada momento y adapta el flujo según el progreso.
- Durante la ejecución se mantiene un **Task Ledger**, que registra objetivos, subtareas, decisiones y estado del proceso.
- Se implementa mediante **MagenticBuilder**, configurando agentes especializados, un **Standard Manager**, callbacks para eventos y ejecutando el workflow con `run_stream()`.
- Es la mejor opción cuando se necesita **planificación dinámica, delegación inteligente y colaboración entre múltiples especialistas**, aunque implica mayor complejidad y costo de ejecución.
