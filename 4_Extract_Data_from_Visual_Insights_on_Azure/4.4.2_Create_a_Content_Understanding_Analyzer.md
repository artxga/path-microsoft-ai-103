# Create a Content Understanding Analyzer

En la mayoría de los casos, los **Analyzers** se crean desde **Content Understanding Studio**, ya que permite diseñarlos visualmente y probarlos fácilmente.

Sin embargo, también es posible crearlos mediante la **API** o el **SDK de Python**, enviando una definición JSON.

---

# ¿Qué es un Analyzer?

Un **Analyzer** define cómo Content Understanding procesará un tipo de contenido.

Está compuesto por:

- Analizador base (`baseAnalyzerId`)
- Configuración
- Esquema de campos (Field Schema)
- Modelos de IA que utilizará

Su objetivo es transformar contenido no estructurado en información estructurada.

---

# Estructura de un Analyzer

```text
Analyzer
│
├── description
├── baseAnalyzerId
├── config
├── fieldSchema
└── models
```

---

# Definición de un Analyzer

Ejemplo de un analizador para tarjetas de presentación:

```json
{
    "description": "Simple business card",
    "baseAnalyzerId": "prebuilt-document",
    "config": {
        "returnDetails": true
    },
    "fieldSchema": {
        "fields": {
            "ContactName": {
                "type": "string",
                "method": "extract",
                "description": "Name on business card"
            },
            "EmailAddress": {
                "type": "string",
                "method": "extract",
                "description": "Email address on business card"
            }
        }
    },
    "models": {
        "completion": "gpt-4.1",
        "embedding": "text-embedding-3-large"
    }
}
```

---

# Componentes del Analyzer

## description

Descripción del propósito del analizador.

```json
"description": "Simple business card"
```

---

## baseAnalyzerId

Define el analizador base sobre el cual se construye el personalizado.

Ejemplos:

- `prebuilt-document`
- `prebuilt-image`
- `prebuilt-audio`
- `prebuilt-video`
- `prebuilt-invoice`
- `prebuilt-receipt`

Ejemplo:

```json
"baseAnalyzerId": "prebuilt-document"
```

---

## config

Configuraciones generales.

Ejemplo:

```json
"config": {
    "returnDetails": true
}
```

Algunas opciones permiten devolver:

- detalles del OCR
- bounding boxes
- información de grounding

---

## fieldSchema

Es la parte más importante.

Aquí se define toda la información que el modelo debe extraer.

Ejemplo:

```json
"fieldSchema": {
    "fields": {

        "ContactName": {
            "type": "string",
            "method": "extract",
            "description": "Name on business card"
        },

        "EmailAddress": {
            "type": "string",
            "method": "extract",
            "description": "Email address on business card"
        }

    }
}
```

Cada campo define:

- tipo
- método
- descripción

---

## models

Especifica los modelos generativos utilizados por Content Understanding.

```json
"models": {
    "completion": "gpt-4.1",
    "embedding": "text-embedding-3-large"
}
```

Normalmente:

| Modelo | Uso |
|---------|-----|
| completion | Extraer o generar información |
| embedding | Indexación, búsqueda semántica y RAG |

# Crear un Analyzer usando Python SDK

## 1. Crear el cliente

```python
from azure.ai.contentunderstanding import ContentUnderstandingClient
from azure.core.credentials import AzureKeyCredential

endpoint = "<YOUR_ENDPOINT>"
credential = AzureKeyCredential("<YOUR_API_KEY>")

client = ContentUnderstandingClient(
    endpoint=endpoint,
    credential=credential
)
```

---

## 2. Definir el Analyzer

```python
analyzer_name = "business_card_analyser"

analyzer_definition = {
    "description": "Simple business card",
    "baseAnalyzerId": "prebuilt-document",
    "config": {
        "returnDetails": True
    },
    "fieldSchema": {
        "fields": {
            "ContactName": {
                "type": "string",
                "method": "extract",
                "description": "Name on business card"
            },
            "EmailAddress": {
                "type": "string",
                "method": "extract",
                "description": "Email address on business card"
            }
        }
    },
    "models": {
        "completion": "gpt-4.1",
        "embedding": "text-embedding-3-large"
    }
}
```

---

## 3. Crear el Analyzer

```python
poller = client.begin_create_analyzer(
    analyzer_name,
    body=analyzer_definition
)

result = poller.result()

print(f"Analyzer created: {result.analyzer_id}")
```

---

# ¿Qué hace begin_create_analyzer()?

La creación es una operación **asíncrona (Long Running Operation)**.

```text
Cliente
    │
    ▼
begin_create_analyzer()
    │
    ▼
Azure comienza la creación
    │
    ▼
Poller
    │
    ▼
poller.result()
    │
    ▼
Analyzer listo
```

El SDK administra automáticamente el polling hasta que finaliza la operación.

# Crear un Analyzer usando la REST API

También puede crearse directamente mediante HTTP.

## 1. Leer el JSON

```python
import json

with open("card.json", "r") as file:
    schema_json = json.load(file)
```

---

## 2. Preparar la petición

```python
import requests

headers = {
    "Ocp-Apim-Subscription-Key": "<YOUR_API_KEY>",
    "Content-Type": "application/json"
}

url = f"{<YOUR_ENDPOINT>}/contentunderstanding/analyzers/business_card_analyser?api-version=2025-11-01"
```

---

## 3. Enviar la petición

```python
response = requests.put(
    url,
    headers=headers,
    data=json.dumps(schema_json)
)
```

---

## 4. Obtener la URL de seguimiento

```python
callback_url = response.headers["Operation-Location"]
```

Azure devuelve una URL para consultar el estado de la operación.

---

## 5. Consultar el estado

```python
result_response = requests.get(
    callback_url,
    headers=headers
)

status = result_response.json().get("status")

while status == "Running":
    result_response = requests.get(
        callback_url,
        headers=headers
    )
    status = result_response.json().get("status")

print("Done!")
```

---

# Flujo usando REST

```text
PUT /analyzers
        │
        ▼
Azure inicia la creación
        │
        ▼
Operation-Location
        │
        ▼
GET callback_url
        │
        ▼
Running
        │
        ▼
Running
        │
        ▼
Succeeded
```

---

# SDK vs REST

| SDK Python | REST API |
|------------|----------|
| Más simple | Más control |
| Maneja automáticamente el polling | Debes hacer polling manual |
| Código más limpio | Útil para cualquier lenguaje |
| Recomendado para aplicaciones Python | Ideal para integraciones HTTP o lenguajes sin SDK |

# Ideas clave

- Un **Analyzer** define cómo se analizará un tipo de contenido.
- Se construye sobre un **baseAnalyzerId** (prebuilt).
- El **fieldSchema** indica qué información extraer, clasificar o generar.
- Se pueden especificar modelos como **gpt-4.1** y **text-embedding-3-large**.
- El SDK simplifica la creación mediante `begin_create_analyzer()`.
- La REST API utiliza una operación de larga duración (LRO), que requiere consultar periódicamente el estado mediante la URL devuelta en `Operation-Location`.
