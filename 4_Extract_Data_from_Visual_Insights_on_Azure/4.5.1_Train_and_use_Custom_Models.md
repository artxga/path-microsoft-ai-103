# Train and Use Custom Models (Azure Document Intelligence)

Cuando los **modelos predefinidos (prebuilt models)** no cubren tus tipos de documentos específicos, puedes entrenar **modelos personalizados (custom models)** para extraer información de tus propios formularios.

Azure Document Intelligence utiliza **aprendizaje supervisado (supervised machine learning)**, donde etiquetas ejemplos de documentos con los campos que deseas extraer y el servicio aprende a reconocerlos en nuevos documentos.

---

# Tipos de modelos personalizados

Azure Document Intelligence ofrece:

- **Custom Template Models**
- **Custom Neural Models**
- **Custom Classification Models**

---

# 1. Custom Template Models

Se basan en un **diseño visual consistente**.

Funcionan mejor cuando todos los documentos tienen prácticamente la misma estructura.

Ejemplos:

- Formularios gubernamentales
- Solicitudes
- Cuestionarios
- Formularios internos

### Características

- Extraen:
  - Key-value pairs
  - Selection marks
  - Tables
  - Regions
  - Signatures
- Entrenamiento rápido (minutos)
- Más de 100 idiomas
- Bajo costo
- Muy alta precisión cuando el layout no cambia

### Ideal para

Documentos con formato fijo.

---

# 2. Custom Neural Models

Utilizan **Deep Learning** y se ajustan con tus documentos etiquetados.

No dependen tanto del layout sino también del contenido del documento.

Funcionan bien para documentos:

- estructurados
- semi-estructurados
- no estructurados

Ejemplos:

- Contratos
- Facturas de distintos proveedores
- Reportes
- Documentos variados

### Características

Además de lo anterior soportan:

- Overlapping fields
- Signature detection
- Table confidence
- Row confidence
- Cell confidence

### Ventajas

Mayor precisión cuando los documentos cambian de formato.

### Desventajas

- entrenamiento más lento
- mayor costo
- consume más recursos

---

# Template vs Neural

|Factor|Template|Neural|
|-------|--------|------|
|Ideal para|Documentos con layout fijo|Documentos con layouts variables|
|Entrenamiento|Muy rápido|Más lento|
|Costo|Bajo|Mayor|
|Precisión|Muy alta si el formato no cambia|Mayor en documentos variados|
|Idiomas|100+|Menos (consultar documentación)|
|Características|Campos, tablas, firmas|Todo lo anterior + confianza avanzada|

---

## Recomendación

Empieza siempre con un **Template Model**.

Solo cambia a un **Neural Model** cuando:

- el layout cambia mucho
- la precisión del Template no sea suficiente

---

# 3. Custom Classification Models

No extraen datos.

Su trabajo es **identificar qué tipo de documento recibió el sistema**.

Ejemplo:

Entrada:

- Factura
- Contrato
- Recibo
- Pasaporte

El clasificador determina cuál es el tipo y luego envía el documento al modelo de extracción correspondiente.

Muy útil cuando existen múltiples tipos de formularios.

---

# Entrenar un modelo personalizado

## Paso 1

Guardar los documentos de entrenamiento en un **Azure Blob Storage Container**.

---

## Paso 2

Para cada documento debes tener:

- documento original
- **ocr.json**
- **labels.json**

Además, un único:

- **fields.json**

### Archivos

**ocr.json**

Resultado del análisis OCR.

---

**fields.json**

Define los campos que deseas extraer.

Ejemplo:

- Nombre
- Fecha
- Total
- Dirección

---

**labels.json**

Relaciona cada campo con su ubicación dentro del documento.

---

## Paso 3

Generar un **SAS URL** del contenedor.

---

## Paso 4

Invocar el:

- Build Model REST API

o el SDK correspondiente.

---

## Paso 5

Obtener el **Model ID** usando:

- Get Model REST API

Ese ID será el que usarás posteriormente para analizar documentos.

---

# Recomendación

Microsoft recomienda utilizar **5 o 6 documentos como mínimo**.

Mientras mayor sea el conjunto de entrenamiento y más variado sea, mejores serán los resultados.

---

# Usar un modelo personalizado

Una vez entrenado, únicamente debes llamar al endpoint de análisis indicando el **Model ID**.

---

## Python

```python
endpoint = "YOUR_DOC_INTELLIGENCE_ENDPOINT"
key = "YOUR_DOC_INTELLIGENCE_KEY"

model_id = "YOUR_CUSTOM_BUILT_MODEL_ID"
formUrl = "YOUR_DOCUMENT"

document_analysis_client = DocumentAnalysisClient(
    endpoint=endpoint,
    credential=AzureKeyCredential(key)
)

task = document_analysis_client.begin_analyze_document_from_url(
    model_id,
    formUrl
)

result = task.result()
```

---

## C#

```csharp
string endpoint = "<endpoint>";
string apiKey = "<apiKey>";

AzureKeyCredential credential = new AzureKeyCredential(apiKey);

DocumentAnalysisClient client =
    new DocumentAnalysisClient(new Uri(endpoint), credential);

string modelId = "<modelId>";
Uri fileUri = new Uri("<fileUri>");

AnalyzeDocumentOperation operation =
    await client.AnalyzeDocumentFromUriAsync(
        WaitUntil.Completed,
        modelId,
        fileUri
    );

AnalyzeResult result = operation.Value;
```

---

# Resultado

La respuesta contiene un objeto **analyzeResult** con:

- texto extraído
- páginas
- tablas
- campos detectados
- valores extraídos
- posiciones dentro del documento
- puntuaciones de confianza

# Composed Models

Un **Composed Model** combina varios modelos personalizados en uno solo.

En lugar de elegir manualmente qué modelo utilizar, Document Intelligence:

1. recibe el documento
2. identifica de qué tipo es
3. selecciona automáticamente el modelo correcto
4. devuelve los datos extraídos

Flujo:

```
Documento
      │
      ▼
Composed Model
      │
      ▼
Clasifica documento
      │
      ├──► Modelo Facturas
      ├──► Modelo Contratos
      ├──► Modelo Recibos
      └──► Modelo Formularios
               │
               ▼
Datos extraídos
```

Es ideal cuando una aplicación procesa múltiples tipos de documentos.

---

# Flujo completo

```
Documentos etiquetados
        │
        ▼
Azure Blob Storage
        │
        ▼
Build Model
        │
        ▼
Obtienes Model ID
        │
        ▼
Analyze Document
        │
        ▼
Campos extraídos
        │
        ▼
Aplicación
```
